#3 fix(rate-limit): make unstable resource fields optional and allow passthrough

fix bug TypeScript ~6 行改动
问题：调用 get_rate_limit 时报错 MCP error -32603，提示 resources.code_scanning_upload 字段必须存在。 排查发现 GitHub API 已不再返回该字段，但 Zod schema 把它标为必填项。
💡 核心修复：将 5 个不稳定字段改为 .optional()，并添加 .passthrough() 容忍未来新增字段。 同时把启动方式从 npx 改为本地构建产物，解决了 npx 安装日志污染 stdout 导致的 JSON-RPC 解析错误。
// 修复前 export const RateLimitSchema = z.object({ resources: z.object({ core: RateLimitResourceSchema, code_scanning_upload: RateLimitResourceSchema, // 必填但已不存在 // ... }), }); // 修复后 export const RateLimitSchema = z.object({ resources: z.object({ core: RateLimitResourceSchema, code_scanning_upload: RateLimitResourceSchema.optional(), // 可选 // ... }).passthrough(), // 容忍额外字段 });

#135 fix(AppHeader): move type filter buttons above search input to avoid tags-dropdown overlay

fix bug React / TypeScript ~20 行改动
问题：在剪贴板列表顶部向上滑动呼出搜索栏时，标签筛选下拉框（tags-dropdown）会覆盖「文本 / 图片 / 文件」等分类筛选按钮，导致用户无法点击筛选。
💡 核心修复：将分类筛选按钮的 DOM 位置移到搜索输入框和标签下拉框之前，确保它们始终在标签下拉框上方可见。
// 修复前：筛选按钮在 tags-dropdown 之后 search-container ├── search-input ├── tags-dropdown (absolute, z-index: 100000) └── type-filter-buttons ← 被遮挡 // 修复后：筛选按钮在 tags-dropdown 之前 search-container ├── type-filter-buttons ← 始终可见 ├── search-input └── tags-dropdown

#1568 fix(bookshelf): improve manual sort drag visual feedback and selection handling

fix bug Kotlin / Jetpack Compose ~4 行改动
问题：用户反馈在书架中选择"手动排序"后，无法拖拽移动书籍位置，且选中书籍时不显示选中状态。 排查发现 sh.calvin.reorderable 库的 ReorderableItem 提供了 isDragging 参数但代码未使用，导致拖拽时无视觉反馈。
💡 核心修复：捕获 isDragging 状态，在拖拽时对书籍项应用 .alpha(0.5f) 半透明效果，提供清晰的拖拽视觉反馈。
// 修复前 ReorderableItem(...) { BookItem( modifier = Modifier.longPressDraggableHandle(...), ... ) } // 修复后 ReorderableItem(...) { isDragging -> BookItem( modifier = Modifier .longPressDraggableHandle(...) .alpha(if (isDragging) 0.5f else 1f), // 拖拽视觉反馈 ... ) }

💡 经验总结

1. 用 AI 辅助开源贡献的实战流程

整个流程可以概括为：Fork 仓库 → 阅读源码定位问题 → AI 辅助分析 → 编写修复 → 本地验证 → 推送分支 → 创建 PR → 标注 AI 使用披露。 关键是 AI 能做代码分析、生成修复方案，但最终验证和决策需要人来把关。

2. 常见坑点

  • schema 过时：API 响应结构随时间变化，Zod/JSON Schema 等校验层需要保持同步，使用 .optional() 和 .passthrough() 可以提高健壮性
  • Npx 启动 MCP Server：npx 的安装日志会写入 stdout，污染 JSON-RPC 协议流，改用本地构建产物可以避免
  • DOM 层叠顺序：绝对定位元素的 z-index 问题往往通过调整 DOM 顺序而非修改 z-index 解决更干净
  • Compose 手势冲突：拖拽和点击事件需要清晰分离，利用库提供的状态参数（如 isDragging）可以改善交互

3. PR 描述规范

好的 PR 描述应该包含：Problem（问题）→ Root Cause（根因）→ Changes（改动）→ Verification（验证）。 加上 Before/After 的代码对比和 AI 使用披露，让维护者一目了然。
